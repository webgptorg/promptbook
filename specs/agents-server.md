# Agents Server — Overview

The **Agents Server** is the web application where **Promptbook agents** live. It lets people create AI
agents by writing them in the [Book language](./book-language.md), and lets end users chat with those
agents through a web UI and a set of HTTP APIs.

It is built on top of the **Promptbook Engine** (a framework-agnostic TypeScript library). The server
adds: persistence, multi-tenant hosting, authentication, a durable chat engine, a web UI, an admin
console, and a large set of integrations. See [Promptbook Engine boundary](./architecture/promptbook-engine.md)
for the exact division of responsibilities.

## What the server provides

1. **Agent authoring & hosting** — Create, edit, organize (in folders), version, clone, import/export,
   and delete agents. Each agent is a piece of [Book language](./book-language.md) source that the server
   parses and compiles into [model requirements](./agent-model-requirements.md). See
   [Agent lifecycle](./agents/lifecycle.md).

2. **Chat** — End users hold conversations with agents. Conversations are persisted, resumable, and
   generated by a **durable background job engine** so a reply keeps generating even if the user closes the
   tab. See [Chat](./chat.md) and [`chat/`](./chat/).

3. **Capabilities/tools** — Agents can use a browser, a search engine, deep research, image generation,
   email, calendars, memory, scheduled wake-ups (timeouts), teammate agents, coding projects, and more —
   all declared with [commitments](./commitments.md) in the agent's source. See
   [Tools & capabilities](./features/tools-and-capabilities.md).

4. **Programmatic access** — An [OpenAI-compatible API](./api/openai-compatibility.md), a
   [Model Context Protocol (MCP)](./api/mcp.md) endpoint, and a first-party
   [management API](./api/management-api-v1.md) for automating agents/folders.

5. **Multi-tenant hosting** — A single deployment and database can host many independent logical servers,
   each keyed by domain. See [Multi-server model](./architecture/multi-server.md).

6. **Administration** — A web admin console to configure the server, inspect usage, manage users, run
   backups, and operate the fleet. See [Admin](./features/admin.md).

7. **Self-hosting** — The same application can run locally or on a VPS via the `ptbk agents-server` CLI,
   backed by SQLite instead of Supabase. See [Self-hosting](./features/self-hosting.md).

## High-level shape

- **Framework:** A Next.js App-Router application (server components, route handlers, and a global
  middleware). All server-side entry points are either **page** routes (UI) or **route handlers** (HTTP
  API). See [Application architecture](./architecture/overview.md).
- **Request routing:** A global [middleware](./architecture/multi-server.md) resolves, per request, which
  logical server (and therefore which database table prefix) the request belongs to, enforces access
  control (IP allow-listing, login), and applies security headers.
- **Persistence:** Either **Supabase/PostgreSQL** (hosted) or **SQLite** (standalone). Tables are
  prefixed per logical server. See [Database](./architecture/database.md) and [`data-model/`](./data-model/).
- **Configuration:** A `Metadata` key/value table (admin-editable at runtime) plus a `ServerLimit` table
  and environment variables. See [Configuration](./architecture/configuration.md).
- **Execution:** Agent chat turns are compiled to model requirements and executed against pluggable LLM
  providers, optionally through git-backed coding-agent runners. See
  [Chat execution model](./chat/execution-model.md) and [Runners](./chat/runners.md).

## The two layers: Engine vs. Server

| | Promptbook Engine (`/src`) | Agents Server (`/apps/agents-server`) |
|---|---|---|
| Nature | Framework-agnostic library | Deployed web application |
| Owns | Book language parsing/compilation, commitments, LLM execution abstractions, tools, collections | Persistence, HTTP APIs, UI, auth, multi-tenancy, durable chat jobs, integrations, admin |
| Reused by | CLI, other apps | This server |

An agent's **behavior** (its "soul") is defined entirely by its Book source and is an Engine concept.
The server is responsible for **storing** that source, **compiling** it in a server context (resolving
references, preparing externals), **executing** chat turns durably, and **presenting** the result.
