# Database

The Agents Server persists everything except large binaries (which go to a CDN/object store — see
[File uploads](../features/file-uploads.md)) in a relational database. Two backends are supported.

## Backends

Selected by `PTBK_AGENTS_SERVER_DATABASE`:

- **`supabase`** (default, hosted) — PostgreSQL accessed through Supabase. Uses `POSTGRES_URL` for SQL/
  migrations and `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY` + `SUPABASE_SERVICE_ROLE_KEY`
  for client access. Row-Level Security is enabled on every table.
- **`sqlite`** (standalone/local) — a single SQLite file (`PTBK_AGENTS_SERVER_SQLITE_PATH`, default
  `.promptbook/agents-server.sqlite`) fronted by a Supabase-compatible shim so the same data-access code
  works unchanged.

Backend detection is centralized (`resolveAgentsServerDatabaseMode`). Data access is obtained through
provider helpers (`$provideSupabaseForServer` / `…ForBrowser` / `…ForEdge` / `…ForWorker`).

## Table prefixing (multi-tenancy)

Every server-owned table is stored under a **per-server prefix** (e.g. `server_Core_Agent`). Code always
refers to tables by their **logical** name and resolves the physical name at runtime:

```
$getTableName('Agent')  →  "<activeServerTablePrefix>Agent"
```

The active prefix is chosen per request by the [middleware](./multi-server.md). The **only** unprefixed
table is the global [`_Server`](./multi-server.md) registry. Throughout these specs, tables are named
without their prefix.

> Migration files use the literal token `prefix_` in table names (e.g. `prefix_Agent`); the migration
> runner substitutes the real prefix for each target server.

## Migrations

- Migration SQL lives in `apps/agents-server/src/database/migrations`, named
  `YYYY-MM-NNNN-description.sql` and applied in filename order.
- A migration runner (`runDatabaseMigrations`) applies pending migrations for a set of prefixes, using an
  **execution lock** (`GenerationLock`-style) so concurrent instances don't collide; it can `skip` when
  another instance is already migrating.
- **Automatic migrations**: on startup, unless disabled, the server migrates the current deployment's
  prefixes (`ensureAutomaticDatabaseMigrations`). Controlled by `SUPABASE_AUTO_MIGRATE` (set to
  `0`/`false`/`no` to opt out). Skipped during builds and tests, and when no connection string is present.
  Runs at most once per prefix per process; a failed attempt is retried on the next load rather than
  poisoning the process.
- Which servers/prefixes a manual migration targets is chosen from the [`_Server`](./multi-server.md)
  registry, filtered by environment (`selectPrefixesForMigration`). Admins can trigger migrations from the
  [servers admin API](../api/admin-api.md).

### Backward-compatibility rule (critical)

Because production and preview servers share one physical database, **migrations must be backward
compatible**: after a newer preview deployment migrates the schema, older server versions must keep
running. Therefore:

- **Add**, don't remove or rename existing columns/tables, and don't change the meaning of existing ones.
- New columns should be nullable or have defaults.
- Older servers may simply not use new features; that is acceptable. New tables/columns are safe.

## Conventions

- Primary keys are usually `BIGINT GENERATED BY DEFAULT AS IDENTITY`; some tables (chats, jobs, timeouts,
  push subscriptions, share targets) use a string `id`.
- Timestamps are `TIMESTAMP WITH TIME ZONE` (`createdAt`, `updatedAt`, and domain-specific ones).
- Soft deletes use a nullable `deletedAt` (agents, folders, memory, wallet).
- Cross-entity references to an agent use `agentPermanentId` → `Agent.permanentId` with `ON DELETE CASCADE`
  (durable), rather than the mutable `agentName`.
- JSON payloads use `JSONB`.

## Seeding

On first run a server can be seeded with core/default agents (`seedCoreAgents`, `seedDefaultAgents`) and
default [`Metadata`](./configuration.md) rows. Several migrations also insert/normalize default metadata
(e.g. `DEFAULT_VISIBILITY`, `SERVER_VISIBILITY`, `DEFAULT_THEME`, `NEW_AGENT_WIZARD`,
`DEFAULT_AGENT_AVATAR_VISUAL`, Shibboleth defaults).
