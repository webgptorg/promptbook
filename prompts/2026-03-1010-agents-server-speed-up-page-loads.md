[🐢⚡] Speed up Agents Server page loads + perceived performance

-   *(@@@@ Written by agent)*
-   Overall loading of the Agents Server pages is perceived as super slow. Do a performance-first audit of the request waterfall + SSR/CSR split and implement improvements that reduce both real and perceived latency.
-   Primary success metrics (track before/after, at least on prod and one staging):
    -   TTFB p95 on main pages (homepage, agent profile, chat, my-chats) <= @@@ ms
    -   LCP p95 <= @@@ ms
    -   INP p95 <= @@@ ms
    -   Total JS transferred on first load <= @@@ KB (gzip)
    -   Server CPU time per request (Next.js) <= @@@ ms
-   First step is measurement (no guessing):
    -   Add lightweight server timing instrumentation (e.g. `Server-Timing` header) for key spans: auth, db, external fetches, rendering, serialization.
    -   Add client-side measurement hooks (Web Vitals) and log to existing logging/analytics pipeline @@@.
    -   Add a developer-only “Perf overlay” (query param like `?perf=1`) showing: route, TTFB, hydration time, number of API calls, bytes downloaded.
-   Identify and remove slow critical path work:
    -   Audit each route’s `getServerSideProps`/server components (or equivalent) for N+1 queries, over-fetching, and unnecessary auth checks.
    -   Ensure “global” data (current user, teams, navigation) is not re-fetched multiple times per navigation.
    -   For pages that don’t need strict SSR, switch to partial SSR + client fetching with skeletons.
-   Data fetching optimizations:
    -   Introduce caching for expensive read-only endpoints (in-memory LRU for single instance + shared cache @@@ if multi-instance), with clear TTLs and invalidation on writes.
    -   Batch/parallelize independent fetches (`Promise.all`) and reduce sequential awaits.
    -   Add pagination / infinite scrolling where lists are heavy (agents list, my-chats, logs) and ensure API supports `limit`, `cursor`.
    -   Add DB indices for the most common filters/sorts discovered during audit.
-   Rendering & bundle optimizations:
    -   Ensure heavy components are lazy-loaded (Monaco editor, graph visualizations, markdown highlighters) and never block initial route render.
    -   Split code by route and by feature; reduce shared “vendor” bundle bloat.
    -   Prefer server-rendered HTML for above-the-fold static content, defer non-critical JS.
    -   Optimize images (agent avatars, OG images): correct sizing, `srcset`, caching headers.
-   UX improvements for perceived speed:
    -   Introduce consistent skeleton/loading states for slow sections (profile header, chat history, my-chats list).
    -   Add optimistic UI for actions (creating chat, sending message, toggles) with clear failure fallback.
    -   Keep navigation responsive: prefetch next likely route(s) and data on hover/idle.
    -   Make “empty” states immediate (render layout + placeholders instantly, stream/fetch content progressively).
-   Guardrails & regression prevention:
    -   Add automated perf checks in CI @@@ (e.g. Lighthouse CI) for key routes and budget thresholds.
    -   Add a “perf budget” document and enforce max bundle size and max API calls per route.
    -   Add tracing for slow requests and alerting thresholds @@@.
-   Out of scope (unless found necessary during audit): full rewrite, switching frameworks, major UI redesign.
-   Implementation should be incremental and safe; each change should be measurable and reversible.
