[🔐] Fix SSRF in `/api/page-preview/*` and `/api/team-agent-profile` endpoints of Agents Server

-   Three additional endpoints accept a caller-supplied URL and fetch it server-side **without** authentication and **without** the SSRF guard ([`assertSafeUrl`](apps/agents-server/src/utils/assertSafeUrl.ts)) that already protects `/api/scrape`:
    -   [`apps/agents-server/src/app/api/page-preview/screenshot/route.ts`](apps/agents-server/src/app/api/page-preview/screenshot/route.ts) — opens an HTTP(S) URL in a headless Playwright browser and returns a PNG screenshot. The only validation is that the scheme is `http:` or `https:`; nothing blocks `http://127.0.0.1`, `http://169.254.169.254` (AWS / GCP / Azure metadata service), or RFC-1918 private ranges. The headless browser also executes JavaScript on the fetched page, magnifying the impact.
    -   [`apps/agents-server/src/app/api/page-preview/check/route.ts`](apps/agents-server/src/app/api/page-preview/check/route.ts) — performs a server-side `fetch` (HEAD with redirect-follow) through [`checkIfUrlCanBeEmbedded`](apps/agents-server/src/utils/iframe/checkIfUrlCanBeEmbedded.ts) with the same lack of validation.
    -   [`apps/agents-server/src/app/api/team-agent-profile/route.ts`](apps/agents-server/src/app/api/team-agent-profile/route.ts) — calls `fetchAgentProfile` against any URL that passes [`isValidAgentUrl`](src/utils/validators/url/isValidAgentUrl.ts). `isValidAgentUrl` only checks that the URL parses, uses `http(s):`, and has no fragment — the file itself comments _"Is it secure to allow pipeline URLs on private and unsecured networks?"_. The endpoint then forwards the server-side internal team access token to the target, so an attacker can also exfiltrate that header.
-   The fix is to apply the existing `assertSafeUrl(url)` guard from [`apps/agents-server/src/utils/assertSafeUrl.ts`](apps/agents-server/src/utils/assertSafeUrl.ts) before any outbound `fetch` / `page.goto`, and to **require authentication** on these endpoints (`getCurrentUser` at minimum). For `/api/team-agent-profile`, also avoid forwarding `createTeamInternalAgentAccessHeaders` to targets that are not on the same origin (the helper already returns an empty header map for cross-origin URLs, but the call must only happen after the URL is asserted safe). The TODO comment `[👣][🧠]` in `isValidAgentUrl` should be resolved by reusing `assertSafeUrl` in callers that hit untrusted networks instead of putting the check inside that pure validator.
-   Do a proper analysis of the current functionality before you start implementing — check how `/api/page-preview/screenshot` and `/api/page-preview/check` are used by the chat iframe-fallback feature added in [2026-06-0340-agents-server-iframe-security.md](prompts/2026-06-0340-agents-server-iframe-security.md) and make sure the fix does not break legitimate previews of public pages. Group all three endpoints into this one PRD because they share the same root vulnerability.
-   Keep in mind the DRY _(don't repeat yourself)_ principle — reuse the existing `assertSafeUrl` helper.
