[🔐] Fix internal worker token and team access token reusing `ADMIN_PASSWORD` (and using a hardcoded fallback) in Agents Server

-   Two distinct internal shared-secret tokens fall back to `ADMIN_PASSWORD` (and other unrelated secrets) when their dedicated environment variable is missing. This is the same class of issue as [`2026-06-0460-security-oauth-state-secret-fallback.md`](prompts/2026-06-0460-security-oauth-state-secret-fallback.md) and [`2026-06-0430-security-session-secret-key.md`](prompts/2026-06-0430-security-session-secret-key.md): one leaked credential cascades into multiple unrelated authentication boundaries.
-   The vulnerability is found in **two places**:
    -   [`apps/agents-server/src/utils/userChat/resolveUserChatWorkerInternalToken.ts`](apps/agents-server/src/utils/userChat/resolveUserChatWorkerInternalToken.ts) — when `PTBK_AGENTS_SERVER_USER_CHAT_WORKER_TOKEN` is not set, the token is derived as `sha256('user-chat-worker:' + (SUPABASE_SERVICE_ROLE_KEY || ADMIN_PASSWORD || NEXT_PUBLIC_SITE_URL || 'promptbook-user-chat-worker'))`. If an operator deploys without explicitly setting this variable (the likely default), the token degrades to `sha256('user-chat-worker:promptbook-user-chat-worker')` — a **publicly known** constant. An attacker who computes that hash can call [`/api/internal/user-chat-jobs/run`](apps/agents-server/src/app/api/internal/user-chat-jobs/run/route.ts) and [`/api/internal/agent-runner-limits/route.ts`](apps/agents-server/src/app/api/internal/agent-runner-limits/route.ts) to drive jobs at will. Even in the "good" path the token is derived from `SUPABASE_SERVICE_ROLE_KEY` or `ADMIN_PASSWORD`, so leaking those leaks the worker token too — and vice versa, an offline brute-force of the worker token reveals the underlying secret.
    -   [`src/commitments/_common/teamInternalAgentAccess.ts`](src/commitments/_common/teamInternalAgentAccess.ts) lines 38–48 — `resolveTeamInternalAgentAccessToken()` chains `PROMPTBOOK_TEAM_AGENT_ACCESS_TOKEN || SUPABASE_SERVICE_ROLE_KEY || ADMIN_PASSWORD`. This token authorizes same-server TEAM calls into private teammate agents, so the same secret-reuse and admin-password-leak concerns apply.
-   The fix is to:
    1.  Require dedicated environment variables for each token (`PTBK_AGENTS_SERVER_USER_CHAT_WORKER_TOKEN` and `PROMPTBOOK_TEAM_AGENT_ACCESS_TOKEN`) and remove the `|| SUPABASE_SERVICE_ROLE_KEY || ADMIN_PASSWORD || ...` chain.
    2.  Generate a cryptographically random value at first boot when the variable is missing and persist it (e.g. via the existing VPS env-var storage), or fail-closed with a clear configuration error in production. **Never** silently fall back to a publicly-known string such as `'promptbook-user-chat-worker'` — that turns the gate into a no-op.
    3.  Update deployment documentation and any `.env.example` files to describe the new variables.
-   Do a proper analysis of the current functionality before you start implementing — read how the existing OAuth state secrets are configured (see the referenced PRDs above) and follow the same pattern so all three "must be a dedicated secret" stories share one approach.
-   Keep in mind the DRY _(don't repeat yourself)_ principle.
-   You are working with the [Promptbook Engine](src) and the [Agents Server](apps/agents-server)
-   Add the changes into the [changelog](changelog/_current-preversion.md)
